# Offer 捕手 · Codex 入职指南

## 项目一句话
Offer 捕手是一个学生求职匹配工具，主流程是简历解析、JD 解析、匹配度计算、优化建议和历史记录。

## 仓库地图
- `backend/`：FastAPI 后端。
- `frontend/`：当前唯一主前端，入口是 `index.html`。
- `docs/`：方案说明、交接文档、提示词设计等详细知识库。
- `start.sh`：一键启动脚本，启动前端静态页和后端 `8888` 端口。
- `deprecated-frontend-nextjs/`、`deprecated-frontend-nextjs-backup/`、`deprecated-empty-frontend-nextjs-shell/`：废弃目录，不作为运行入口。

## 先信什么
- 优先信代码、启动脚本、依赖文件、测试结果和实际运行状态。
- 不要直接信 README 或交接文档里的版本描述；如有冲突，以真实代码为准。
- 不要把“计划支持”当成已完成能力。

## 运行原则
- 后端入口：`backend/app/main.py`
- 后端依赖：`backend/requirements.txt`
- 前端入口：`frontend/index.html`
- 前端通过 `python3 -m http.server` 启动，不依赖 Node 构建

## 设计原则
- 先保留闭环：上传/解析/匹配/建议/历史记录优先于装饰性功能。
- 优先简洁、可维护、可验证的实现。
- 中文产品语境优先，少用过度技术化表达。
- 先做能跑通、能解释、能复查的改动，再做美化。

## 编码规范
- 改后端先看 `backend/app/services/` 是否已有可复用实现。
- 改前端优先动 `frontend/index.html`，避免重新引入废弃前端路线。
- 保持改动小而明确，尽量不把单次修改扩成架构重写。
- 涉及启动方式、端口、入口时，同步检查 `start.sh`。

## 风格偏好
- 优先简洁、可维护代码。
- 以真实可运行结果为准，不做“看起来像完成”的假闭环。
- UI 以清晰信息层级和求职场景理解成本为先。

## 禁止事项
- 不要恢复废弃的 Next.js 前端作为主入口。
- 不要把文档里的示例、规划、愿景直接写成已完成事实。
- 不要只改 README 不改代码，也不要只改代码不核对运行入口。
- 不要为了“更完整”而引入与当前单前端方案冲突的新技术栈。

## 审查标准
- 代码是否符合当前真实架构。
- 启动脚本、端口和入口是否一致。
- 改动后是否能本地跑通或至少清楚说明未验证原因。
- 文档是否和实现一致，是否存在夸大或过时描述。
- 是否保持了现有中文产品表达和简洁风格。

## 详细知识库
- 架构、历史和功能边界：`docs/项目交接文档.md`
- 产品方案、流程和技术选型：`docs/方案说明.md`
- 简历匹配提示词与分析原则：`docs/简历匹配分析提示词设计.md`

## 额外提醒
- 当前仓库以“单前端 + FastAPI”作为真实运行基线。
- 如果要改端口、入口或运行方式，优先同步修正脚本和文档。
- 如果发现文档和代码冲突，先修事实，再修表述。
