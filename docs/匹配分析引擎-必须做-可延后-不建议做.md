# 匹配分析引擎：必须做 / 可延后 / 不建议做

## 一句话结论

匹配分析引擎真正要做的，不是堆很多 AI 名词，而是把这三件事做对：

1. `把 JD 和简历拆成可比较的结构`
2. `先召回最相关证据，再做逐条判断`
3. `把判断结果映射成可执行的改写和补位动作`

---

## 一、必须做

这些是产品从“前端 demo”变成“真实可运行后端系统”时最核心的部分。

### 1. JD 结构化解构

必须做。

目标：

- 把 JD 从一整段文本，拆成稳定的结构化要求

最低要输出：

- `硬性门槛`
- `核心技能`
- `核心业务场景`
- `角色层级`
- `关键 requirement 列表`

为什么必须做：

- 不先拆 JD，后面的匹配、gap、改写都会漂
- 匹配分析是否“像真人招聘官”，第一步就取决于有没有真正看懂 JD

### 2. 简历 / 原子库结构化解构

必须做。

目标：

- 把简历和原子库拆成可比对的经历单元

最低要输出：

- `项目 / 经历标题`
- `场景`
- `动作`
- `结果`
- `工具 / 技能`
- `职责层级`

为什么必须做：

- 不拆开，模型只能做整段模糊判断
- 你要做逐条 requirement 分析，就必须先把候选人经历拆成可核证对象

### 3. requirement 级匹配分析

必须做，而且是核心中的核心。

目标：

- 对 JD 的每一条关键 requirement 分别判断

至少要支持：

- `已匹配`
- `证据不足`
- `经历缺失`

为什么必须做：

- 用户最终关心的不是总分，而是“哪条行、哪条不行、为什么”
- 这决定产品是“关键词匹配器”还是“招聘官分析器”

### 4. 混合召回逻辑

必须做，但可以先做轻量版。

至少包含：

- `关键词召回`
- `语义相似度召回`

召回对象：

- 当前简历相关片段
- 原子库中最相关的卡片

为什么必须做：

- 不先召回相关证据，LLM 很容易全局乱判
- 你后面要做推荐补位原子，也必须先有召回能力

### 5. 结构化 JSON 输出

必须做。

后端至少要稳定输出：

- `requirement_checks`
- `gap_type`
- `resume_evidence`
- `reason`
- `fix_strategy`
- `recommended_atoms`

为什么必须做：

- 前端三分类展示依赖它
- 左分析、中简历、右原子库联动也依赖它

### 6. 分析到改写的映射

必须做。

至少支持 3 类动作：

- `保留并前置`
- `改写补证`
- `推荐插入 / 替换原子`

为什么必须做：

- 没有这一步，系统就只是点评器
- 用户要的是“改出来”，不是“被分析完”

## 二、可以做，但先简化

这些方向是对的，但当前阶段不建议做太重。

### 1. 混合检索的工程化基础设施

可以做，但先轻量。

现在建议：

- 关键词匹配
- embedding 相似度
- 内存级 top-k 排序

先不要急着上：

- Milvus
- Pinecone
- 重型向量数据库架构

原因：

- 你现在更需要验证“召回逻辑是否真的提升分析质量”
- 不是先追求商用级检索基础设施

### 2. STAR 分析

可以做，但先轻量。

建议做法：

- 不要做教科书式完整 STAR
- 重点拆出：
  - `场景`
  - `动作`
  - `结果`
  - `个人贡献`

原因：

- 目的是提升比对和改写，不是做面试培训系统

### 3. 动态权重打分

可以做，但先模板化。

建议做法：

- 按岗位类型设固定权重模板

例如：

- AI 产品岗：业务场景 35，项目相关性 35，工具技能 20，表达 10
- 工程岗：硬技能 40，项目相关性 30，复杂度 20，表达 10

先不要做：

- 完全开放式 LLM 自动动态加权

原因：

- v1 重点应该是逐条判断准不准
- 不是把打分系统做得特别复杂

### 4. 原子库自动推荐

建议做，而且重要，但先做“推荐”，不要一步到位“自动替换”。

建议先实现：

- 推荐最适合补某条 gap 的 2-3 条原子
- 解释为什么推荐它
- 让用户确认后再插入或替换

原因：

- 这样更稳，也更符合当前产品阶段

### 5. ownership / 证据强度判断

可以做，但作为辅助维度。

可以分析：

- `参与 / 协助 / 配合` -> 弱 ownership
- `主导 / 独立负责 / 设计并落地` -> 强 ownership

但建议命名成：

- `职责层级判断`
- `证据强度判断`

不要直接叫：

- `含水量分析`
- `吹嘘检测`

原因：

- 现在做太“判真假”，容易误伤，也容易显得武断

## 三、不建议现在做

这些不是永远不做，而是当前做了很容易把系统做重、做虚、做偏。

### 1. 重型 Agent 编排

现在不建议。

包括：

- 多 Agent 协作
- planner / executor / critic 大编排
- 复杂 Agent orchestration

原因：

- 当前更需要的是稳定 schema、稳定 prompt、稳定召回和稳定输出
- 单模型 + 结构化流程，往往比多 Agent 更稳

### 2. 商用级向量数据库基础设施

现在不建议。

除非马上要支持：

- 海量候选人库
- 海量 JD 检索
- 多租户大规模在线服务

否则先别做重型基础设施。

### 3. 关键词 stuffing

现在明确不建议。

不建议做法：

- 为了过 ATS，硬塞行业词
- 把没做过的岗位语言缝进简历

建议替代原则：

- `真实证据显性化`

也就是：

- 把用户真实做过的经历用更贴岗位语言的方式说清楚
- 但不把没做过的东西包装成做过

### 4. 过重的总分体系

现在不建议花太多精力。

原因：

- 总分是辅助信息，不是核心价值
- 你现在更该投入的是：
  - requirement 逐条判断
  - 证据不足 vs 经历缺失
  - 原子推荐
  - 改写映射

### 5. 强判“用户是否在吹牛”

现在不建议。

建议替代成：

- `这条证据强不强`
- `这条表述像不像真实做过`
- `这条内容是否经得起追问`

原因：

- “真假判定”很难做准
- 但“证据强度判断”更可行，也更符合产品定位

## 四、推荐的 v1 后端流程

建议把 v1 收成下面这 5 步：

### 第 1 步：JD 结构化

输出：

- 硬门槛
- 核心技能
- 业务场景
- 角色层级
- requirement 列表

### 第 2 步：简历 / 原子库结构化

输出：

- 项目拆解
- 技能标签
- 场景标签
- 结果表达
- ownership 强度

### 第 3 步：混合召回

召回：

- 当前简历最相关片段
- 原子库最相关卡片

方式：

- 关键词匹配
- embedding 相似度

### 第 4 步：LLM 精排

对每条 requirement 输出：

- 是否匹配
- 证据在哪里
- 为什么不足
- 是证据不足还是经历缺失
- 推荐哪条原子补位

### 第 5 步：改写映射

输出：

- 哪些保留
- 哪些改写
- 哪些替换 / 插入原子
- 前端 diff 所需结果

---

## 五、真正的开发优先级

### P0

- JD 结构化
- 简历 / 原子库结构化
- requirement 级分析
- 结构化 JSON 输出

### P1

- 混合召回
- 原子推荐
- 改写映射

### P2

- 更细的岗位类型权重
- 更强的 ownership 判断
- 面试追问辅助

### 暂缓

- 多 Agent
- 重型向量数据库
- stuffing
- 复杂分数系统

---

## 六、最终判断

如果要把这个产品从“前端动效”做成“真正可落地的后端 AI 系统”，最关键的不是：

- Agent 名字够不够高级
- 向量库是不是商用级
- 分数图表够不够复杂

而是这 3 个问题有没有解决：

1. `系统有没有真正看懂 JD`
2. `系统有没有逐条核验证据`
3. `系统有没有把分析结果变成能直接改简历的动作`

只要这三件事做对，产品就成立。
