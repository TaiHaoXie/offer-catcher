# 简历匹配分析提示词设计

## 目标

这套提示词不是普通的“简历打分 Prompt”，而是要模拟更接近真实大厂校招招聘官的判断方式，同时对求职者真正有帮助。

目标有两个：

1. 让分析结果更像真实招聘判断，而不是泛泛而谈的 AI 夸奖。
2. 让求职者看完之后，知道自己到底差在哪、哪些能改写补、哪些必须补真实经历、下一步该怎么做。

## 核心设计思路

### 1. 从“是否优秀”改成“是否匹配这个岗位”

很多简历产品的问题是：

- 只会夸候选人背景不错
- 只会罗列关键词命中率
- 不会区分“人不错”和“对这个 JD 合不合适”

真实招聘官并不是抽象地看你优不优秀，而是看：

- 这个岗位的硬门槛你过不过
- 这个岗位最关键的能力你有没有证据
- 你的经历是否足够相关
- 你的表达是否让人相信你真的做过

所以提示词里要强制模型先拆 JD，再逐项核对。

### 2. 必须区分“能力缺失”和“证据缺失”

这是求职产品里非常关键的一点。

如果候选人没有写清楚，不等于候选人不会。

因此输出里要明确区分：

- `capability_gap`: 真不会，或者现有经历无法支撑
- `evidence_gap`: 可能做过，但简历没写出来
- `experience_gap`: 岗位需要相关经历，简历里确实没有
- `expression_gap`: 表达弱、没有量化、前1/3不抓人

只有这样，用户才会觉得产品“懂我”，而不是一味打低分。

### 3. 输出不要停在分析，要直接进入行动

真正好用的产品不能只说：

- 你缺少 A
- 你缺少 B
- 建议补充项目经验

这类建议太空。

更好的输出应该直接告诉用户：

- 先改哪一段
- 改这段的目标是什么
- 用什么方法改
- 哪些 24 小时内就能改
- 哪些 7 天内可以通过补一个小项目/案例弥补
- 哪些必须中长期补真实能力

所以输出结构里加入了：

- `rewrite_priorities`
- `action_plan.within_24_hours`
- `action_plan.within_7_days`
- `action_plan.longer_term`

### 4. 要像大厂招聘官，而不是像“鼓励型 AI”

为了让结果更专业，系统 Prompt 里明确约束：

- 不要鸡汤
- 不要无证据夸奖
- 不要把“没写出来”直接当“不会”
- 不要编造经历和成果
- 结论先行，标准严格，但态度友好

这会显著提升用户对结果的信任感。

## 最有价值的输出字段

### 1. `one_sentence_verdict`

一句话说透：

- 这个人和这个岗位到底匹不匹配
- 最关键原因是什么

这很像真实招聘官在简历筛选时的第一判断。

### 2. `requirement_checks`

这是整份分析最核心的部分。

每个要求都必须写清楚：

- 这是什么要求
- 属于硬门槛还是核心能力还是加分项
- 判断结果是什么
- JD 证据是什么
- 简历证据是什么
- 为什么这么判断

这会让结果特别“像人事在过简历”。

### 3. `gaps`

用户最关心的是：

- 我到底差在哪
- 这个 gap 严不严重
- 能不能靠改写解决

因此 `can_be_fixed_by_rewrite` 非常关键，它直接决定用户接下来是改简历，还是补经历。

### 4. `application_strategy`

一个好产品不只是说“你不匹配”，还要告诉用户：

- 现在要不要投
- 更适合投哪些岗位
- 哪些岗位先别硬投

这部分会让产品从“分析工具”升级成“投递决策助手”。

## 推荐的产品级调用方式

### 单轮版本

第一轮先输出完整招聘官分析：

- 总评
- JD 拆解
- 逐项核对
- gap 分类
- 改写优先级
- 行动计划
- 投递策略

适合首次分析。

### 双轮版本

第二轮继续做“补 gap 辅导”：

可以基于第一轮结果继续追问：

- 帮我改写第 1 段项目经历
- 给我一个 7 天补齐这个岗位 gap 的计划
- 如果我没有这段实习，怎么用项目替代证明能力
- 帮我针对这个 JD 重排简历前 1/3

这样体验会比一次性大而全更强。

## 这套 Prompt 为什么会让用户觉得“这是我用过最好的”

因为它同时满足了四件事：

1. 像真实招聘官，而不是像关键词检测器。
2. 对用户公平，能区分“不会”和“没写清楚”。
3. 不只指出问题，还给有优先级的行动方案。
4. 不只服务“改简历”，还服务“决定该不该投、该投什么”。

真正让用户记住的，不是分数，而是：

- “它真的看懂了我为什么过不了”
- “它告诉我哪些问题今天就能改”
- “它没有骗我，也没有空泛安慰我”
